Q1

Quel smartphone est compatible avec AR-SIGHT ?
Tout iPhone 12 Pro ou supérieur (LiDAR intégré) et les Android compatibles ARCore. La liste complète est disponible dans la documentation.

Non. Le scan prend 5 à 10 minutes par pièce, avant la fermeture des cloisons. C'est la seule étape ajoutée à son processus actuel.

Le Hub utilise un laser de classe 2, la classification la plus courante des niveaux mesureurs. Aucun EPI spécifique requis. Certifié CE.

Page 4 — Preuve & Conformité

H1

Chaque scan a une valeur légale.

Section Horodatage

Horodatage certifié.
Chaque scan est associé à un timestamp RFC 3161, non modifiable. Date, heure, coordonnées GPS du chantier. Opposable devant un tribunal.

Section Intégrité

Intégrité des données : SHA-256.
Un hash cryptographique est généré à la création de chaque fichier scan. Toute modification ultérieure du fichier invalide le hash. Preuve d'intégrité irréfutable.

Section RGPD

Conformité RGPD.
Données hébergées en France (OVH Cloud, Paris). Accès sur authentification. Durée de conservation configurable. DPA disponible. Registre des traitements tenu à jour.

Section Laser

Hub Laser — Sécurité classe 2.
Conformité IEC 60825-1. Puissance <1mW. Longueur d'onde : 650 nm (rouge visible). Aucun EPI requis. Certification CE. Fiche de sécurité disponible sur demande.

Q6. AR-SIGHT fonctionne-t-il sans connexion internet ?
Les données de scan sont mises en cache localement. La visualisation AR fonctionne hors-ligne une fois le modèle téléchargé. La synchronisation reprend dès que la connexion est rétablie.

Q7. Peut-on intégrer AR-SIGHT à notre logiciel de gestion de chantier ?
L'API est disponible avec la formule Hub. Les connecteurs natifs pour les principaux logiciels (Batigest, Onaya) sont en cours de développement.

Q8. Quelle est la précision du scan LiDAR ?
La précision du scan iPhone LiDAR est de ±5 mm à 3 m. Le Hub Laser affine la projection à ±3 mm à 2 m de la surface.

Article — As-Built vs BIM

H1 article : As-Built vs BIM : quelles différences, quels usages sur chantier ?

Introduction : Le BIM (Building Information Modeling) est conçu en phase de conception. L'as-built est la réalité du chantier une fois construit. Ces deux notions sont souvent confondues — au détriment de la traçabilité.

Partie 1 : Le BIM : un modèle de conception, pas une preuve.
Le BIM prévu et le bâtiment réel divergent dès le premier coup de marteau. Modifications, aléas, adaptations terrain : aucun BIM ne capture la réalité du chantier en temps réel.

Partie 2 : L'as-built : la vérité du terrain.
Un relevé as-built capture l'état réel après construction. Avec AR-SIGHT, ce relevé est automatique, horodaté et consultable en AR. Il remplace avantageusement les plans papier annotés à la main.

Conclusion : AR-SIGHT ne remplace pas le BIM. Il comble le fossé entre le plan prévu et ce qui est vraiment dans les murs. Pour les corps de métier en phase de finition, c'est cette couche de réalité qui compte.

G. Checklist "Ready for Dev"

Responsive & Layout

✓
Breakpoints définis
Mobile-first. 390px / 768px / 1024px / 1200px. Testé sur iPhone SE, iPhone 14, iPad, 1440px desktop.
✓
Nav hamburger fonctionnelle
Transition slide-down douce, fermeture au clic extérieur, aria-expanded géré.
✓
Grilles responsive
grid-4 → grid-2 → grid-1 selon breakpoints. Pas de scroll horizontal.
✓
Images optimisées
Format WebP, srcset défini, lazy loading (loading="lazy"), dimensions explicites (width/height) pour éviter le CLS.

Accessibilité (WCAG 2.1 AA)

✓
Contraste texte
#1D1D1F sur #FFF : 19.7:1 ✓ · #6E6E73 sur #FFF : 5.7:1 ✓ · #0A84FF sur #FFF : 3.1:1 (à réserver aux éléments décoratifs + texte large uniquement)
✓
Focus visible sur tous les éléments interactifs
outline 2px #0A84FF, offset 3px. Testé clavier tab/shift+tab/enter/espace.
✓
ARIA labeling
aria-label sur hamburger, aria-expanded sur accordéon FAQ, aria-describedby sur erreurs formulaire, aria-live="polite" sur toast.
✓
Ordre de lecture logique
DOM = ordre visuel. Pas d'inversion CSS qui casse le tab order.
✓
Textes alt sur toutes les images
Images décoratives : alt="". Images informatives : description précise.
✓
prefers-reduced-motion respecté
Toutes les animations CSS désactivées si la préférence est active. Implémenté dans le CSS.
✓
Landmark ARIA
<header>, <nav>, <main>, <section>, <footer> présents. role="region" avec aria-labelledby sur les sections sans balise native.

Performance (Lighthouse ≥ 90)

✓
Pas de JS bloquant au-dessus du fold
Tous les scripts : defer ou en bas de body. Pas d'import synchrone inutile.
✓
CSS critique inline
Above-the-fold CSS inliné dans <head> si performance critique. Le reste en async.
✓
Font loading optimisé
font-display: swap. Preload des 2 familles principales. Variable fonts si disponible.
✓
LCP < 2.5s
Image hero : fetchpriority="high" + preload. Pas de lazy loading sur l'image hero.
✓
CLS < 0.1
Toutes les images ont width/height explicites. Pas de fonts qui font sauter le layout.
✓
Pas de dépendances JS inutiles
Vanilla JS uniquement en v1. Pas de jQuery, lodash, ou framework lourd pour ce site vitrine.
